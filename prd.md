# 产品需求文档 (PRD)：统一推送平台 (UniPush)

## 1. 产品概述

### 1.1 项目背景
在日常研发与运维中，各个业务系统常常需要发送告警或通知（如对接企业微信 Bot、钉钉 Bot）。目前存在以下痛点：
1. **管理混乱**：各个项目分别去申请渠道的 Webhook Key，散落在不同的代码仓库或配置文件中。
2. **重复开发**：每个项目都需要单独编写对接企微、钉钉的代码，报文格式各不相同。
3. **安全隐患**：第三方 Webhook 链接直接暴露在业务代码中，一旦泄露难以统一撤回。
4. **排障困难**：消息漏发时，难以排查是业务系统没发、网络问题，还是被第三方平台限流。

### 1.2 产品目标
建立一个统一的消息推送网关（中转站）。业务侧只需对接平台统一的 API，平台负责适配不同的目标渠道并完成分发。实现**渠道统一管理、权限统一管控、日志统一追踪**。

---

## 2. 核心名词解释

| 名词 | 解释 |
| :--- | :--- |
| **渠道 (Channel)** | 实际接收消息的第三方载体，如：某个钉钉群机器人、企微群机器人、自定义 Webhook 地址等。 |
| **渠道配置 (Channel Config)** | 包含渠道的类型以及必要的凭证（如 Webhook URL、Secret）。由管理员或拥有权限的用户维护。 |
| **推送密钥 (Push Key / Token)** | 平台分配给调用方的唯一凭证。业务系统通过该 Key 向平台发送消息。 |
| **绑定关系 (Binding / Router)** | 一个 Push Key 可以绑定一个或多个 Channel。消息到达 Key 后，会路由到所有绑定的渠道。 |

---

## 3. 业务流程与架构

### 3.1 核心数据流
1. **配置阶段**：管理员在后台配置渠道 -> 用户新建 Push Key -> 用户将 Key 绑定至所需渠道。
2. **发送阶段**：业务系统携带 Push Key 请求平台 API -> 平台校验鉴权并持久化 -> 平台根据绑定关系异步转换报文 -> 平台请求第三方渠道 -> 记录第三方返回结果。

### 3.2 逻辑架构图 (示意)
```text
[业务系统 A] --(API/Key1)--> |          | ---> 转换器 A ---> [钉钉群 Bot]
[业务系统 B] --(API/Key2)--> | 统一网关 | ---> 转换器 B ---> [企微群 Bot]
[脚本工具 C] --(API/Key3)--> |          | ---> 转换器 C ---> [自定义 Webhook]
```

## 4. 功能需求详细说明

### 4.1 管理后台 (Admin Console)

#### 4.1.1 渠道管理模块
* **新增渠道**：支持选择渠道类型（钉钉、企微、飞书、自定义 Webhook），并输入第三方提供的 URL 及鉴权 Secret。
* **渠道列表**：展示所有渠道的名称、类型、状态（启用/禁用）、创建时间。
* **编辑/删除**：支持修改渠道配置，支持软删除。
* **渠道连通性测试**：点击“测试”按钮，平台向该渠道发送一条默认测试消息，以验证配置是否有效。

#### 4.1.2 平台大盘 (Dashboard)
* 统计并展示今日请求量、发送成功数、发送失败数。

### 4.2 用户侧/开发者侧 (User Console)

#### 4.2.1 密钥 (Key) 管理
* **创建 Key**：输入业务名称（如：订单系统告警），系统自动生成一段随机且唯一的 Push Key（如 UUID）。
* **配置路由规则（绑定渠道）**：用户可从已有的渠道列表中，勾选一个或多个作为该 Key 的发送目标。
* **重置 Key**：当业务侧密钥泄露时，可一键重置，旧 Key 立即失效。
* **停用 Key**：暂停该 Key 的接收。

#### 4.2.2 消息日志与追踪 (Message Logs)
* **日志列表**：展示每条消息的 `Message_ID`、接收时间、使用的 Key、目标渠道、最终发送状态（成功/失败）。
* **日志详情**：点击可查看客户端原始请求报文，以及第三方渠道返回的**原始响应（Response Body）**，方便定位被拦截或拒收的原因。

### 4.3 核心路由与发送引擎 (System Engine)

* **标准报文转换**：对外暴露统一的数据结构，引擎根据目标渠道类型，动态转译为目标 JSON 格式（Adapter 适配器模式）。
* **异步处理**：接口接收请求后，写入内部队列（如 Redis/MQ）并立即返回 `Message_ID`，由后台 Worker 异步消费，避免阻塞业务系统。
* **一键多发**：如果一个 Key 绑定了 3 个渠道，引擎需生成 3 个子任务分别发送。

## 5. 非功能性需求

### 5.1 安全与风控
1. **频控限制 (Rate Limiting)**：
   * **单 Key 级别**：限制单 Key 每分钟最多发送 N 条 (用户可设置)，超出直接拒绝（HTTP 429）。
   * **单 Channel 级别**：限制单渠道每分钟最多发送 N 条（管理员创建渠道可设置），超出时进入队列排队等待，不直接丢弃。
2. **防滥用**：平台禁止发送体积过大（如超过 10KB）的文本。

### 5.2 性能与高可用
1. **接口响应时间**：网关接收消息的 API (入队操作) 响应时间要求 P99 < 50ms。
2. **数据保留策略**：消息日志数据量大，仅保留最近 30 天的日志（.env可设置），过期自动清理。

## 6. API 设计规划 (草案)

为了保证极简接入，需提供 RESTful 接口和简单的 GET 接口。

### 6.1 极简 GET 调用 (适合 Shell 脚本)
* **URL**: `GET /api/v1/send/{PushKey}?title=标题&content=正文内容`
* **Response**:
  ```json
  {
    "code": 200,
    "msg": "success",
    "data": { "message_id": "msg_abc123456" }
  }
  ```

### 6.2 标准 POST 调用 (适合业务系统)
* **URL**: `POST /api/v1/push`
* **Headers**: `Authorization: Bearer {PushKey}`
* **Body**:
  ```json
  {
    "title": "生产环境异常告警",
    "content": "### 异常详情 \n NullPointerException...",
    "type": "markdown" // text, markdown
  }
  ```
